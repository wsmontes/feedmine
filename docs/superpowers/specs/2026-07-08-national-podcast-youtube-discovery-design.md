# National Podcast & YouTube Discovery — Design

**Date:** 2026-07-08
**Branch:** `worktree-feed-discovery`
**Status:** Approved design, pending implementation plan

## Context

The `scripts/feed_discovery` pipeline discovers national RSS feeds per country by
running anchored DuckDuckGo searches (`site:.<cctld>`), autodiscovering `<link
rel="alternate">` feeds on the result pages, verifying liveness, and emitting one
OPML + one report per country.

This model works well for text categories (News, Blogs, Business) but is
**structurally wrong** for two of the 26 categories:

- **YouTube** — `build_queries` produces `canal youtube rss site:.bo`. The
  `site:.bo` operator *forces* Bolivian domains, so it can never surface a
  `youtube.com/...` channel. In the bolivia reference run the YouTube bucket was
  filled with `itseller.bo` (an e-commerce site).
- **Podcasts** — `podcast rss site:.bo` only finds `.bo` blogs that mention the
  word "podcast", not the actual podcast RSS (which lives on Spotify / Apple /
  Anchor). The bolivia bucket got `carloshugomolina.com.bo/feed/` (a personal
  blog), not real podcasts.

A podcast's RSS and a YouTube channel's RSS live **off the ccTLD**, so
"national = URL ends in `.bo`" cannot be the nationality test for these two
categories. They need dedicated discovery sources.

## Goals

- Discover genuinely national **podcasts** and **YouTube channels** for a country
  and emit their real feed URLs into the existing per-country OPML.
- Keyless: no API keys or quota management, so it scales to 100+ countries in a
  batch script.
- Quality over coverage: strict nationality filtering (the user's explicit
  choice), consistent with the pipeline's "no international leakage" mission.
- Reuse the existing pipeline (liveness verify, OPML emit, report) unchanged.

## Non-goals / out of scope

- **Dedup canonicalization** (`www`, `/feed/` ≡ `/rss/` ≡ `rss.xml`, OJS
  variants, cross-category dupes) — the bolivia run showed ~31% duplicate feed
  variants. Tracked as a **separate follow-up spec (workstream A)**.
- **Categorization precision** for the other 24 DDG categories (institutional
  `.gob`/`.edu` feeds landing in wrong buckets) — **separate follow-up spec
  (workstream B)**.
- Spotify discovery (no public RSS), video-URL → channel resolution, YouTube
  topic/genre classification.

## Locked decisions

| Decision | Choice |
| --- | --- |
| Data sources | Keyless: **iTunes Search API** (podcasts) + **DDG search + channel scrape** (YouTube) |
| Bucketing | **Format buckets** — podcasts → `Podcasts`, channels → `YouTube` — with **genre/topic as outline metadata** |
| Nationality | **Strict for both** — podcast requires iTunes `country == ISO3`; channel requires About-page `country == country display name` |
| Integration | Integrated into `process_country` (workstream Approach 1), single OPML + report per run |

### Feasibility validation (done during design)

- iTunes `GET /search?term=bolivia&country=bo&entity=podcast` returned 8 results,
  **all with `feedUrl`**, `country=BOL`, and localized `primaryGenreName`
  (`Política`, `Historia`, `Educación`, …).
- `https://www.youtube.com/feeds/videos.xml?channel_id=UC…` returns valid Atom.
- A channel `.../about` page exposes `"country":"United States"` /
  `"United Kingdom"` inside `aboutChannelViewModel`, and `channelId":"UC…"` is
  present in the page HTML — both extractable keyless.

## Architecture & integration

New submodule `scripts/feed_discovery/sources/` with two files, each exposing the
same async interface:

```
sources/podcasts.py →  async discover(country, session, cfg) -> list[Candidate]
sources/youtube.py  →  async discover(country, session, cfg) -> list[Candidate]
```

`process_country` routes categories: when iterating categories, `Podcasts` and
`YouTube` are dispatched to these functions; the other 24 categories keep the
existing `build_queries` + DDG + autodiscovery path. All candidates join the
**same list**, and the downstream stages are reused unchanged:

- dedup against `existing_urls`
- async liveness verify
- comment-feed title drop
- `emit_opml`
- `report.summarize`

Output remains one `<slug>.opml` + one `report.md` per run.

## Data model changes

1. **`Country`** gains two fields:
   - `iso2` — iTunes storefront param. Derived from `cctld` with a small override
     map for the handful of ccTLD ≠ ISO2 cases (e.g. `uk → gb`).
   - `iso3` — strict podcast nationality filter (iTunes result `country` is
     alpha-3, e.g. `BOL`). Sourced from an `alpha2 → alpha3` table in `data/`.
   - `countries.json` is regenerated by `data/build_countries.py`.
2. **`Candidate`** gains `genre: str = ""` (iTunes `primaryGenreName` for
   podcasts; empty for YouTube).
3. **`emit_opml`** writes an optional `category="<genre>"` attribute on the feed
   `<outline>` when `genre` is non-empty (extra attribute; the app ignores
   unknown attributes). `candidates_to_opml_map` extends its per-feed tuple from
   `(title, url)` to `(title, url, genre)`.

## Component: podcast discovery (`sources/podcasts.py`)

1. **Seed terms** = `country.name` + `country.native_name` (if different) + each
   city in `country.cities`. Reuses existing metadata; no new config.
2. Per term, call the iTunes Search API:
   `GET https://itunes.apple.com/search?term=<term>&country=<iso2>&entity=podcast&limit=50`
3. **Strict nationality filter:** keep only results where
   `result["country"] == country.iso3`.
4. **Dedup** within the run by `feedUrl` (fallback `collectionId`).
5. Emit `Candidate(url=feedUrl, category="Podcasts", title=collectionName,
   genre=primaryGenreName, national=True,
   national_reason="itunes country==iso3")`. The `feedUrl` flows into the normal
   liveness verify with the rest.

**Rate limit / cache:** apply `cfg.delay` between iTunes calls (~20 req/min Apple
guidance); cache each term's JSON at `cache/itunes/<slug>/<term>.json`, respecting
`--fresh` (same pattern as the search cache).

**Coverage limitation (accepted):** keyless iTunes has no "list all podcasts from
country X" endpoint — only term search. Coverage is therefore bounded by podcasts
that match the seed terms *and* have `country == iso3`. This favors precision over
recall, consistent with the strict choice. Broadening seed terms later is additive
(no refactor).

## Component: YouTube discovery (`sources/youtube.py`)

1. **Seed queries (DDG)** anchored on the country, targeting channels:
   `site:youtube.com <name>`, `site:youtube.com <city>` per city,
   `<name> canal youtube`. Reuses `search.search()` (DDG region + cache + delay).
2. **Extract channel references** from result URLs — forms
   `youtube.com/channel/UC…`, `/@handle`, `/c/Name`, `/user/Name`. Video URLs
   (`/watch?v=`) are ignored in v1. Dedup channel URLs.
3. **Resolve + read About:** one fetch of `…/about` per channel (browser UA)
   extracts, in one pass: `channelId` (regex `"channelId":"UC…"`),
   `"country":"<name>"` (from `aboutChannelViewModel`), and the channel title.
4. **Strict nationality filter:** keep only channels where
   `country == country.name` (display-name match; small alias set if needed). A
   channel with no country set is dropped.
5. **Build RSS:** `https://www.youtube.com/feeds/videos.xml?channel_id=<UC…>` →
   `Candidate(url=rss, category="YouTube", title=<title>, genre="",
   national=True, national_reason="youtube about country==name")`. Dedup by
   `channel_id`. Flows into the normal liveness verify.

**Genre:** keyless YouTube gives no reliable channel genre, so YouTube candidates
carry `genre=""` (genre metadata is populated for podcasts only). No invented
categories.

**Cache / fragility:** cache per-channel extraction at
`cache/youtube/<slug>/<channel>.json` (respects `--fresh`); `/about` fetches run
concurrently bounded by `cfg.concurrency` + connector `limit_per_host=5`. The
`channelId` / `country` extractor is a small, fixture-tested unit — if YouTube's
HTML changes, YouTube discovery yields 0 and **fails safe** without contaminating
other categories. A per-channel fetch failure skips that channel and logs.

## Error handling & rate limiting

- **iTunes / DDG / YouTube fetch failures:** caught per request; the offending
  term/channel is skipped and logged, never aborting the country run.
- **Empty results:** a source returning zero candidates is valid (strict filter);
  the category simply has no feeds, as with other empty categories today.
- **Rate limiting:** `cfg.delay` between iTunes and DDG calls; per-host connector
  cap for `/about` fetches; all responses cached and `--fresh`-aware to avoid
  re-hitting external services across runs.

## Testing strategy

Unit tests with recorded fixtures — **no live network in tests**:

- `sources/podcasts.py`: fixture iTunes JSON (mix of `country==iso3` and other
  countries) → asserts strict filter, dedup, genre capture, Candidate shape.
- `sources/youtube.py`: fixture channel `/about` HTML (with and without a country
  field) + fixture DDG results → asserts channelId extraction, country match /
  reject, RSS URL construction, dedup by channel_id.
- `emit_opml`: asserts the optional `category` attribute is written when genre is
  present and omitted when empty.
- `process_country` routing: asserts `Podcasts` / `YouTube` categories dispatch to
  the new sources and other categories keep the DDG path (with the network layer
  stubbed).
- Data model: assert every registry country resolves a valid `iso2` and `iso3`.

Existing 42 tests must stay green.

## Rollout

1. Implement data-model changes + regenerate `countries.json`.
2. Implement `sources/podcasts.py` + tests.
3. Implement `sources/youtube.py` + tests.
4. Wire routing into `process_country` + `emit_opml` metadata.
5. Re-run bolivia (`--country bolivia --fresh`) and evaluate the Podcasts /
   YouTube buckets against this spec's quality bar before running more countries.
